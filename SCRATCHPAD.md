**Decisions made**
- Impersonation token: signed with the Web Crypto API (`crypto.subtle`, HMAC-SHA256) using `NEXTAUTH_SECRET` as the signing key, rather than reusing NextAuth's own JWT encoding — this was the only way to make it verifiable from `middleware.ts`, which runs on the Edge runtime and can't load Prisma's native query engine (same constraint noted in the Phase 0 report for the suspended-company check). The token lives in a separate `impersonation` httpOnly cookie; the real NextAuth session is never touched, so exiting impersonation is just deleting one cookie.
- Audit attribution while impersonating: `getTenantContext()` returns `{ companyId, userId, role }` for the *impersonated* user (so any future business record's `createdBy` reflects the account that "created" it) plus a separate `impersonatedBy` field with the real super admin's id, for `AuditLog` rows. No tenant-data actions exist yet that would use this (Phase 1 has none), but the pattern is now established — **future phases writing `AuditLog` from inside impersonated sessions must use `ctx.impersonatedBy ?? ctx.userId` as the actor, not `ctx.userId` alone.**
- "Most active companies" (ARCHITECTURE.md §4, abbreviated in PHASES.md's Phase 1 bullet) interpreted as "most recently active" (sorted by the same `lastActivity` computation used elsewhere), since there's no sales data yet to rank by transaction volume — this will read more meaningfully once Phase 5+ adds invoices.
- Branding preview and company-scoped audit trail were included on the company detail page even though PHASES.md's own Phase 1 bullet list only says "stats, users, suspend/activate, reset admin password" — ARCHITECTURE.md §4's fuller page description for `/super/companies/[id]` explicitly lists "branding preview" and "audit trail", and CLAUDE.md says never contradict ARCHITECTURE.md. **Known issues / tech debt**
- Real bug found and fixed during testing: Next.js automatically re-renders the *current* route after any Server Action resolves, regardless of whether the caller also calls `router.refresh()`. This meant clicking "Exit" while impersonating briefly re-rendered `/dashboard` as a plain (non-impersonating) SUPER_ADMIN session before the client-side `router.push("/super")` landed, and `getTenantContext()` used to `throw` in that case — surfacing a server error. Fixed by having `getTenantContext()` `redirect("/super")` instead of throwing when a SUPER_ADMIN has no active impersonation; this is a graceful, expected transient state, not an application error. Low severity (was cosmetic/log noise, never blocked the actual exit flow), but worth knowing if a similar "SUPER_ADMIN mid-impersonation-exit" state shows up elsewhere.
- Real bug found and fixed: the dashboard greeting and the company shell's sidebar user name both used to read `getServerSession().user.name` directly, which is the *real* super admin's name while impersonating, not the impersonated user's. Fixed by always resolving the display name via `prisma.user.findUnique({ where: { id: ctx.userId } })` (works identically whether impersonating or not, since `ctx.userId` is already the effective user). **Any future page that shows a user's name must resolve it the same way (via `getTenantContext()` + a User lookup), not via `getServerSession()` directly**, or it will misattribute the name during impersonation.
- Impersonation token has a fixed 1-hour expiry with no renewal — a long super-admin support session doing more than an hour of work as a company admin would need to re-click "Login as" after expiry. Acceptable for now; revisit if support sessions run long in practice.
- `/super/audit` has no pagination or filtering — just the latest 200 rows. Fine at current volume; will need a real filter/paginate UI once the audit table grows.
- No dedicated empty state yet for `/super/companies` when there are zero companies beyond the DataTable's generic "No companies yet" text — matches the design system's empty-state pattern loosely but doesn't use the specified "centered icon in a 20px-radius surface square" treatment from ARCHITECTURE.md §5. Minor, cosmetic.

**Notes for next phase**
- `lib/actions/super-admin.ts` and `lib/queries/super-admin.ts` are the established pattern for this kind of work: queries return plain serializable data for Server Components, actions are `"use server"` functions guarded by `requireSuperAdmin()`/(future) an analogous `requireRole()` helper, called directly from client components (not `<form action>`) so they can return typed data (e.g. the temp password) for the caller to use — mirrors Phase 0's `LoginForm` pattern.
- `components/ui/data-table.tsx` is generic and ready to reuse for Phase 3's product table, Phase 4's customers table, etc. Remember: column definitions with cell renderers (JSX, icons) must be defined *inside* a `"use client"` file that itself renders `<DataTable>` — passing a `columns` array built in a Server Component throws the same "functions can't cross the RSC boundary" error documented in the Phase 0 report. `CompaniesTable`/`UsersTable`/`AuditTable` are the reference pattern: the page (Server Component) fetches data and passes plain rows; the table component (Client Component) owns the column defs.
- `components/ui/dialog.tsx` (Radix-based) is now available for any future one-time-reveal or confirmation UI.
- Never wrap a call to a "use server" function that might call `redirect()` in your own `try/catch` — `redirect()` throws a sentinel Next.js needs to catch itself, and an outer catch will swallow it and show a false "error". `startImpersonation`/`exitImpersonation` deliberately don't call `redirect()` themselves for this reason; the client component calls `router.push()` after `await`-ing the action instead.
- Local dev super admin credentials are in `.env` (not committed); demo companies now include four "Test Solar …" companies created during this session's Playwright testing, left in the local DB alongside the Phase 0 fixtures (`demo-solar`, `suspended-solar`) — harmless, local-only, fine to ignore or reset next session.

- **`createInvoice`** (`lib/actions/invoices.ts`, one `$transaction`): validates the customer (if any) belongs to the company, snapshots each line's product name/price into `InvoiceItem`, decrements stock per line via a conditional `updateMany` (`WHERE stockQty >= qty`, sequential `for` loop so oversell-checking and multiple lines for the same product both see each other's effects, matching Phase 3's atomic-adjustment pattern) and writes a `StockAdjustment(SALE, qtyChange: -qty, refId: invoice.id)` per line, writes `LedgerEntry(INVOICE, debit: total)` only if a customer is attached, writes `Payment` + `LedgerEntry(PAYMENT, credit: amountPaidNow)` for any amount paid now, derives `status` from `paidAmount` vs `total` (UNPAID / PARTIAL / PAID), and writes an `AuditLog` row — all inside the same transaction, so a mid-loop oversell throw rolls back everything (no invoice row, no stock change, no ledger entry).
- **`recordInvoicePayment`**: reads the invoice inside its own transaction and updates it with an optimistic conditional `updateMany` (`WHERE paidAmount = <value just read>`) — if a concurrent payment already changed `paidAmount`, the WHERE no longer matches, `count === 0`, and it throws "updated elsewhere — please retry" instead of silently overpaying. Same technique as the stock-decrement guard, applied to the payment race the task called out specifically. Rejects amounts exceeding the remaining balance. If the caller is a CASHIER, also requires `invoice.createdBy === ctx.userId` (defense in depth on top of the page-level restriction below).
- **Business rule, enforced server-side (not just in the UI):** a walk-in invoice (`customerId` empty) must be paid in full at creation — there's no ledger to carry a walk-in balance on, so an unpaid walk-in sale would silently lose the receivable. Both `PosScreen` and `InvoiceForm` block submission client-side with an inline message, and `createInvoice` re-checks the same rule itself before writing anything, per CLAUDE.md's "UI hiding is not security."

**Decisions made**

- **`updateQuotationStatus`**: manual transitions among `DRAFT`/`SENT`/`ACCEPTED`/`EXPIRED`/`REJECTED` (a plain `<select>` on the detail page, changes take effect immediately on `onChange`, no separate save step). `CONVERTED` is not manually selectable and blocks any further status change once reached.
- **`convertQuotationToInvoice`**: reverse-derives each line's `lineDiscount` from the quotation's already-stored `lineTotal` (`qty × unitPrice − lineTotal`, floored at 0) so the resulting invoice reproduces the exact prices the quote promised, then calls the **real, unmodified `createInvoice`** from `lib/actions/invoices.ts` — same oversell check, same stock decrement, same numbering lock, same `LedgerEntry(INVOICE)` write, all running for the first time at conversion since the quote itself never touched any of it. `amountPaidNow: 0` always (task requirement: converted invoices are always UNPAID). On success, marks the quote `CONVERTED` and stores `Quotation.convertedInvoiceId` in one follow-up `$transaction` (kept deliberately separate from `createInvoice`'s own transaction — see Decisions).
- **"Linked both ways" without a schema change**: `Quotation.convertedInvoiceId` is the one stored link (quote → invoice, already in the Phase 0 schema). The reverse direction (invoice → quote) is a query, not a column: `getConvertedFromQuotation(companyId, invoiceId)` in `lib/queries/quotations.ts` looks up `WHERE convertedInvoiceId = invoiceId`. The invoice detail page calls this and shows "Converted from QUO-0001" when it finds one; the quotation detail page shows "Converted to INV-0001" from its own stored field. Both directions navigable, zero migration.

Shows "QUOTATION" + valid-until date instead of a paid/balance-due breakdown (quotes aren't paid).

**Decisions made**
- **Quotation items always reference a real product; no free-text/custom line items.** `QuotationItem.productId` is nullable in the schema (room for a future non-inventory line), but PHASES.md's own Phase 6 bullet says "saved customer **or free-text name**" — the free-text option is explicitly about the *customer*, not items. Since conversion is required to go through the real `createInvoice` (which hard-requires a real `productId` per item for stock/oversell purposes), building a UI path that produces an unconvertible quote would be a half-finished feature. `quotationItemInputSchema` is literally `invoiceItemInputSchema` re-exported, so this is enforced at the type level, not just left to convention.
- **Converting a quote requires a saved customer** (`Quotation.customerId`, not just `customerNameFree`). `createInvoice` always requires `amountPaidNow ≥ total` for a walk-in (Phase 5A rule — no ledger to carry an unpaid walk-in balance on), and a converted quote is always `amountPaidNow: 0` per this phase's own requirement — those two rules are only reconcilable if the invoice has a real customer. A free-text-only quote can still be created, viewed, PDF'd, and shared; it just can't one-click-convert, and the UI explains why instead of showing a broken/confusing button. This is a genuine business-rule consequence of Phase 5A's existing walk-in guard, not an arbitrary restriction.
- **`convertQuotationToInvoice` calls `createInvoice` and lets it run its own separate transaction**, rather than inlining everything into one giant transaction — literal reading of the task ("through the **existing** createInvoice transaction path"). The follow-up `quotation.update` (status → CONVERTED) + audit log run in their own small transaction immediately after. Narrow edge case this leaves open: if the process crashes between the invoice committing and that follow-up transaction running, the invoice exists but the quote isn't marked CONVERTED yet — recoverable (nothing is lost, no double-charge, no phantom stock), just a staleness window, and unavoidable without either merging transactions (which the task's wording argues against) or making `createInvoice` quote-aware (which would be a Phase 5A modification for a Phase 6 need).
- **Quotation PDF styles are duplicated, not extracted into a shared module with the invoice PDF**, even though they're currently identical. Chose isolation over DRY here specifically because Phase 5B's invoice PDF is already tested and working — refactoring it to import shared styles would touch a previously-verified file for a purely cosmetic reason. ~30 lines of `StyleSheet.create` duplication is a small price for that isolation; documented per the Notes-for-next-phase precedent this same trade-off shows up if Phase 7 adds a purchase-order PDF.
- **Returns lock the `Invoice` row (`FOR UPDATE`), not a new counter or field**, to make the "sold minus already-returned" cap race-free under concurrent return submissions on the same invoice. This is the same *tool* (row-lock-inside-transaction) the task told me to reuse for quotation numbering, applied to a different *problem* (a multi-row aggregate read-then-write needs a critical section, not a single-column sequence) — extending an established, already-correct pattern rather than inventing a new concurrency primitive. Verified the returnable-quantity re-read happens *after* the lock is acquired, not before, so a just-committed concurrent return is always accounted for.
- **A rejected over-return leaves no trace beyond the failed attempt** — no partial `Return` row, no partial stock/ledger writes — because the whole cap-check-and-write sequence is one transaction that throws before any `tx.return.create` runs. Confirmed via Playwright (see Checklist).
- **Return accounting is maximally conservative, per ARCHITECTURE.md §3's literal wording ("adjusts invoice paid/status logic conservatively")**: `createReturn` contains zero writes to the `Invoice` table. `Invoice.paidAmount`/`status` stay exactly as `createInvoice`/`recordInvoicePayment` last left them, forever, regardless of returns. A cash refund (if the business actually hands cash back) is a real-world action the `Return.note` field can record — the system never automates or infers it. This was the more conservative of two readings available (the other being "reduce paidAmount by the returned amount"); chose it because Phase 5A's SCRATCHPAD notes explicitly warned not to reuse `recordInvoicePayment`'s status logic here, and CLAUDE.md's financial hard rules favor the reading that can't silently misstate what was actually paid in cash.

**Errors found and fixed this phase**
- **Real bug: `loading.tsx` silently breaks `notFound()`'s HTTP status code.** Next.js App Router wraps a route segment *and every descendant route* in a `<Suspense>` boundary wherever a `loading.tsx` exists (unless a more specific nested `loading.tsx` overrides it). The initial HTTP response streams — and its status code commits — before a suspended descendant page finishes rendering, so if that descendant later calls `notFound()`, the page's *content* correctly ends up showing "404 - This page could not be found," but the response's actual HTTP status code is already 200. Caught via `tests/tenant-isolation.spec.ts` failing "Expected 404, Received 200" after `loading.tsx` was first added to detail routes; confirmed with a standalone diagnostic script capturing `response.status()` directly (not just the rendered text) before touching any test file. Fixed by removing `loading.tsx` from every route with a `notFound()`-capable descendant, at both the detail level and the parent list level (the cascade isn't depth-limited): `inventory/`, `inventory/[id]/`, `customers/`, `customers/[id]/`, `purchases/`, `purchases/[id]/`, `invoices/`, `invoices/[id]/`, `quotations/`, `quotations/[id]/`, `(super)/super/`, `(super)/super/companies/[id]/`. Kept only where nothing downstream calls `notFound()`: `dashboard/`, `expenses/`, `reports/`, `settings/`, `suppliers/`. Worth remembering for any future route added to this app: a `loading.tsx` is not free, and its blast radius covers everything nested beneath it, not just its own segment.
- **Real bug: shared `DataTable` rendered both desktop and mobile layouts into the DOM simultaneously**, using Tailwind's `hidden md:block` / `md:hidden` to CSS-hide whichever didn't apply. Playwright's `getByText(..., { exact: true })` matches DOM content regardless of CSS visibility, so any exact-text assertion against a value that appeared in both variants hit a strict-mode "resolved to 2 elements" failure (surfaced first in `tests/reports.spec.ts`). Fixed by adding a `useIsMobile()` hook and rendering exactly one variant, never both — the correct fix regardless of the test failure, since shipping two copies of every table row to every page load was already wasted DOM/bytes.
- **Real bug carried over from Phase 7: the tamper-token Playwright test had a ~6.25% chance of not actually testing anything.** `tests/invoice-outputs.spec.ts` flipped the *last* character of a base64url-encoded signed token to assert a tampered link 404s. A 32-byte HMAC-SHA256 signature has 2 leftover bytes in its final 3-character base64 group, so the last character only encodes 4 real bits (the other 2 are always-zero padding) — flipping it to "A" or "B" can land on a character sharing the same top 4 bits as the original whenever the original character was itself "A," decoding to the *identical* signature byte and leaving the token still valid. Verified empirically with a 200,000-trial standalone script: 6.215% collision rate with the old approach, 0.000% after switching to flipping the *second-to-last* character instead (which fully participates in real bits). This explains an intermittent, previously-unexplained flake noted but not root-caused in the Phase 7 report.
- Diagnosed (not a code bug): a throwaway console-error-monitoring script used for this phase's own Lighthouse/console-error checklist item initially reported 5 "Failed to fetch RSC payload... Falling back to browser navigation" console errors. Traced to the script itself calling `page.goto()` with full absolute URLs back-to-back from an already-loaded page, which can trigger Next.js's client router into a failed RSC prefetch before falling back to a normal navigation — a benign degradation of the script's own navigation pattern, not an app defect. Confirmed by switching the script to relative-path navigation via a proper `browser.newContext({ baseURL })` (matching how every real Playwright test and `loginAs()` helper in this suite navigates): 0 console/page errors across the full authenticated route set.

**Known issues / tech debt**
- **Production builds redact Server Action error messages — a real, pre-existing, app-wide UX gap, not fixed this phase.** Running the suite against `next build && next start` instead of `next dev` (as an experiment while investigating dev-mode flakiness) revealed that Next.js 14.2.x redacts thrown `Error` messages from Server Actions by default in production builds, replacing them with a generic message client-side — regardless of try/catch on the calling side. - **Collapsed nav links have no accessible name** (icon-only, no `aria-label`/`title`) — also pre-existing since Phase 0, not introduced by Task A. `tests/sidebar.spec.ts` had to target the collapsed nav link by `href` rather than `getByRole("link", { name: ... })` because of this. Flagging for a future accessibility pass; out of scope for "reduce sidebar width and enlarge the logo," which is all Task A asked for.
- **Real, environment-specific finding this session, documented for future test-running sessions**: this sandbox's local `next dev` process measurably degrades over a sustained session — CPU/memory usage climbs (observed 46–54% CPU *while idle*, ~2GB+ RSS) after roughly 15–20 minutes of continuous operation under repeated Playwright runs, and once degraded, an otherwise-healthy full suite sees double-digit spurious failures (`page.goto` timeouts, "Target page, context or browser has been closed") with the run taking 2–3× longer than normal — all while Postgres and disk/memory at the OS level stayed healthy. Restarting the dev server (`rm -rf .next` + fresh `pnpm dev`) immediately before each full-suite run consistently resolved it — two fully clean 39/39 runs were achieved this way, alongside a couple of 37–38/39 runs with a single isolated failure in a pre-existing, this-phase-untouched spec file (audit-log, purchases, tenant-isolation, quotations — never the same file twice), consistent with this project's already-documented single-request-slowdown flakiness from the Phase 7/8 reports. This dev-server-degradation pattern is new information beyond what those earlier reports captured and is worth a restart-before-suite habit in any future long test-running session here.
- No pagination on the import preview table (loads all rows from the 500-row cap into the DOM at once) — acceptable at that ceiling, same accepted trade-off as every other list page in this app.
- The "Payment details" PDF block always renders in the same fixed position (after the totals summary, before the footer note) regardless of how many fields are set — with all six filled it's a compact box, never overflows the page at realistic invoice lengths, but wasn't stress-tested against an invoice with an unusually large number of line items pushing the summary further down the page.
